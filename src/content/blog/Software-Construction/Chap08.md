## 微服务架构基础

### 微服务定义

Martin Fowler 对微服务的典型定义可以概括为：

**微服务架构是一种将单个应用程序开发为一组小型服务的方法。每个服务运行在自己的进程中，基于轻量级机制通信，围绕业务功能构建，独立部署，并且可以由不同技术栈实现。**

关键点：

- **业务功能边界**
  - 服务按业务能力拆分，而不是按技术层拆分。
  - 例如订单服务、用户服务、支付服务。
- **独立性**
  - 独立进程。
  - 独立部署。
  - 独立维护。
  - 服务之间松耦合。
- **轻量通信**
  - HTTP/REST。
  - gRPC。
  - 消息队列。
  - 避免 SOAP 这类重量级技术栈。
- **技术异构性**
  - 不同服务可使用不同语言、框架、数据库。
  - 但技术栈多样性要受控。

### 核心设计原则

| 原则 | 含义 |
|---|---|
| **高内聚、低耦合** | 服务内部功能完整，服务之间减少频繁依赖 |
| **单一职责与数据自治** | 服务聚焦特定业务责任，并维护自己的数据 |
| **允许技术异构，但避免混乱** | 可自主选型，但不能让维护成本失控 |
| **微服务规模受控** | 服务规模应能被小团队快速开发、测试、部署 |

**复习提示**：单一职责不是“一个服务只做一个函数”，而是一个服务围绕一个清晰业务能力形成完整闭环。

### 密码管理示例

密码管理微服务可以包含：

- 创建密码。
- 修改密码。
- 检查密码。
- 恢复密码。
- 密码有效性检查。
- 密码数据库备份与恢复。
- 完整性检查与修复。
- 消息处理。
- 故障处理。
- 数据一致性管理。

这些功能虽然不止一个操作，但都围绕 **密码管理** 这个高内聚业务能力，因此适合放在同一个服务中。

## DDD 与微服务划分

### DDD 的作用

**领域驱动设计（DDD）** 能解决微服务拆分中的几个核心问题：

- 服务边界如何划分。
- 数据如何分类与隔离。
- 服务之间如何协调。
- 什么模型在什么范围内有效。

DDD 中本章重点关注：

- **限界上下文（Bounded Context）**
- **聚合根（Aggregate Root）**

## 限界上下文

### 基本定义

**限界上下文（Bounded Context）** 是特定业务领域内保持术语、业务概念、业务规则和领域模型语义一致的逻辑边界。

通俗理解：

- 它是一个独立的业务语义边界。
- 边界内部术语含义统一。
- 边界内部业务规则自洽。
- 领域模型只在这个边界内有效。
- 一旦跨越边界，同一个名词可以有完全不同含义。

**一句话**：出了这个边界，同一个名词可以不是同一个东西。

### “用户”的多重含义

| 上下文 | “用户”的含义 | 核心关注 |
|---|---|---|
| **电商会员上下文** | 会员等级、积分、收货地址、优惠券 | 忠诚度、消费能力 |
| **订单上下文** | 下单人、手机号、配送信息 | 履约、交付 |
| **风控上下文** | 行为画像、风险等级、黑名单标识 | 风险评估、安全 |

如果强行共用一个“用户”模型，会导致：

- 模型膨胀。
- 业务逻辑交织。
- 系统耦合严重。
- 后续维护困难。

### 限界上下文解决的问题

- **解决业务概念歧义**
  - 在边界内统一语言，减少沟通误解。
- **划分职责与变更范围**
  - 一个上下文就是一个业务变更单元。
- **成为微服务的天然前身**
  - 理想情况下，一个限界上下文对应一个微服务。

```mermaid
flowchart TD
  A["领域 Domain<br/>完整业务生态"] --> B["子域 Subdomain<br/>垂直业务拆分"]
  B --> C["限界上下文<br/>语义与模型边界"]
  C --> D["微服务<br/>技术落地形态"]
```

### 与微服务的关系

正确拆分方式：

- 先根据业务识别限界上下文。
- 再将限界上下文映射为微服务。
- 服务边界由业务语义决定，而不是由技术层决定。

错误方式：

- 按技术层拆分。
- 按通用功能强行拆分。
- 忽略业务边界。
- 最终容易形成 **分布式单体**。

### 如何识别限界上下文

可以按以下步骤：

1. **梳理整体业务域**
   - 画出完整业务流程。
   - 例如注册、浏览商品、下单、支付、发货、售后、积分。
2. **拆分业务子域**
   - 按业务职能、组织架构、业务流程拆分。
3. **找歧义术语，划语义边界**
   - 找同名不同义。
   - 找只在局部生效的规则。
4. **判定内聚与耦合**
   - 经常一起变更、数据高度依赖、规则互相约束的内容适合放一起。
   - 团队不同、发布节奏不同、领域独立、模型差异巨大的内容应拆开。
5. **固化上下文间映射关系**
   - 同步接口调用。
   - 异步事件总线。
   - 消息队列。
   - 防腐层。

### 向微服务映射

| 规则 | 含义 |
|---|---|
| **一对一原则** | 一个限界上下文对应一个微服务，适合中小型项目 |
| **允许合并，禁止拆分** | 小上下文可合并，完整上下文不要强拆 |
| **数据库边界隔离** | 一个微服务对应独立数据库实例或独立表，禁止跨上下文直接连表 |
| **防腐层保护** | 外部模型不能直接侵入本上下文核心模型 |

**防腐层（Anti-Corruption Layer）** 的作用是适配和转换外部模型，让内部领域模型保持语义纯净。

## 聚合根

### 基本定义

**聚合（Aggregate）** 是一组关联领域对象的集合，被视为一个数据修改单元。

**聚合根（Aggregate Root）** 是聚合的根实体，是聚合对外暴露的唯一入口点。

聚合根负责：

- 控制外部访问。
- 强制执行业务规则。
- 维护数据一致性和完整性。

通俗理解：

**聚合根就是聚合内部状态的守门员。**

### 核心作用

| 作用 | 说明 |
|---|---|
| **维护业务规则** | 保证关键业务不变量始终为真 |
| **定义事务边界** | 聚合内所有修改应在一个原子事务中完成 |
| **简化对象关联** | 外部只引用聚合根，内部对象由根管理 |
| **集中控制访问** | 所有内部状态修改必须通过聚合根完成 |

### 与限界上下文的关系

| 概念 | 边界类型 | 解决问题 | 范围 |
|---|---|---|---|
| **限界上下文** | 语义边界 | 在特定业务语境中，概念是什么意思 | 较大，划分业务领域和职责 |
| **聚合** | 一致性边界 | 一次事务中哪些对象必须一起修改 | 较小，划分数据修改单元 |

不同限界上下文的聚合根之间通常通过 **领域事件（Domain Event）** 协作，实现最终一致性。

```mermaid
flowchart LR
  A["订单聚合根"] --> B["发布 OrderPaidEvent"]
  B --> C["支付上下文订阅"]
  B --> D["物流上下文订阅"]
  B --> E["生产上下文订阅"]
```

### 设计原则

- **设计小聚合**
  - 避免性能瓶颈和并发冲突。
- **引用其他聚合时只引用 ID**
  - 降低耦合，避免加载整个对象图。
- **聚合根是唯一可变入口**
  - 内部对象修改必须通过聚合根方法。
- **聚合根拥有独立生命周期**
  - 内部对象依赖聚合根存在。

### 如何识别聚合根

核心问题：

**为了完成这个业务操作，哪些数据必须被原子性地更新，才能保证业务规则不被破坏？**

识别步骤：

1. 找出业务操作和不变量。
2. 聚类需要一起原子修改的相关对象。
3. 选择能代表整个对象集群的核心对象作为聚合根。
4. 在聚合根上定义对外业务方法，封装内部状态。

### 订单聚合根示例

订单聚合根的核心不变量：

**订单总金额必须等于所有订单项的单价 × 数量之和。**

设计要点：

- 以 **Order** 为聚合根。
- **OrderItem** 和 **Address** 是聚合内对象。
- 外部只能通过 `pay`、`addItem` 等方法修改订单。
- 总额计算方法应由聚合内部控制。
- 支付成功后发布 `OrderPaidEvent`。

## 照片打印系统案例

### 业务子域

照片打印系统可以拆成五个核心子域：

| 子域 | 说明 |
|---|---|
| **用户域** | 用户注册、登录、权限、认证 |
| **订单域** | 照片上传、规格选择、订单生成和状态流转 |
| **支付域** | 多渠道资金结算、退款、支付流水 |
| **生产域** | 打印任务调度、设备状态、耗材库存 |
| **物流域** | 地址管理、物流单号、配送状态追踪 |

### 限界上下文与微服务映射

| 限界上下文 | 职责 | 微服务 |
|---|---|---|
| **用户上下文** | 注册、登录、权限管理、身份认证 | User Service |
| **订单上下文** | 照片上传、规格选择、订单生成 | Order Service |
| **支付上下文** | 支付结算、多渠道适配 | Payment Service |
| **生产上下文** | 打印任务调度、设备监控、耗材管理 | Production Service |
| **物流上下文** | 地址、物流单号、配送状态 | Logistics Service |

### 跨上下文协作

协作方式包括：

- **API 网关路由**
  - 移动应用统一通过网关访问微服务。
- **事件驱动**
  - 订单支付成功后发布“订单已支付”事件。
  - 打印服务和物流服务异步启动后续流程。
- **共享内核**
  - 例如全局唯一订单 ID 作为共享标识。

### 聚合根示例

| 聚合根 | 核心数据 | 核心操作 |
|---|---|---|
| **User** | 用户基础信息、认证凭证、个人偏好 | 注册、登录、第三方账户绑定 |
| **Order** | 订单信息、打印规格、收件地址 | 创建订单、修改参数、更新状态 |
| **Payment** | 支付流水、渠道、金额 | 发起支付、查询状态、退款 |
| **PrintResource** | 设备状态、耗材库存、打印队列 | 查询打印机、执行任务、扣减库存 |
| **Delivery** | 物流信息、承运商、物流单号 | 生成物流单、更新状态、查询轨迹 |

  B --> C["Payment 聚合根处理支付"]
  C --> D["发布 OrderPaidEvent"]

## 微服务通信机制

### 轻量级协议

微服务倾向选择轻量协议。

原因：

- 微服务追求简单性。
- 需要高性能通信。
- 服务之间要松耦合。
- 要支持跨平台兼容。

传统 XML + SOAP 的问题：

- XML 冗长。
- 解析性能较差。
- SOAP 协议过重。
- 耦合度较高。
- 不适合轻量化场景。

### RESTful 与 gRPC

| 特性 | RESTful | gRPC |
|---|---|---|
| **通信对象** | 对外 API，如移动端、第三方系统 | 微服务内部通信 |
| **契约特征** | 松散契约，适合频繁迭代 | 强契约，适合稳定接口 |
| **性能** | JSON/HTTP，文本格式，中等性能 | ProtoBuf/HTTP/2，二进制格式，高性能低延迟 |
| **跨平台难度** | Web 天然兼容，浏览器可直接调用 | 需客户端库，浏览器支持较弱 |
| **典型场景** | API 网关与前端交互、第三方支付 | 内部服务高频数据校验 |

gRPC 的局限：

- 需要提前定义 `.proto` 文件。
- 接口变更要重新生成代码并同步服务。
- 灵活性低于 RESTful。
- 浏览器原生支持弱。
- 对外暴露常需网关转成 HTTP/RESTful。

### 协议选择

| 场景 | 选择 |
|---|---|
| **对外接口** | RESTful，跨平台、易调试 |
| **内部稳定高性能通信** | gRPC，强类型、低延迟 |
| **内部接口频繁变化** | RESTful，松散耦合 |
| **历史系统集成** | API 网关做协议转换，兼容 SOAP 等旧接口 |

### RESTful 示例

照片打印系统基础 URL：

- `https://print-service.com/api/v1`

订单资源示例：

- `GET /api/v1/orders/123`：获取订单 ID 为 123 的订单。
- `GET /api/v1/orders/user/456`：获取用户 ID 为 456 的订单列表。

订单响应体通常包含：

- `orderId`
- `userId`
- `createdAt`
- `media`
- `quantity`
- `price`
- `status`
- `photoUrl`
- `size`
- `deliveryAddress`

## 服务治理关键技术

### 服务注册与发现

微服务实例通常动态扩缩容，地址也会变化，因此需要 **服务注册与发现**。

核心机制：

- **服务注册**
  - 主动注册：服务启动时向注册中心提交地址和元数据。
  - 被动注册：外部组件监测服务状态并更新注册中心。
- **服务发现**
  - 客户端发现：客户端从注册中心获取地址并自行负载均衡。
  - 服务端发现：通过 API 网关或负载均衡器转发请求。
- **注册中心**
  - 维护服务目录。
  - 通过心跳或主动探测做健康检查。
  - 动态更新服务列表。

典型工具：

- Consul。
- Eureka。
- Nacos。
- Kubernetes Service。
- Istio。

```mermaid
flowchart LR
  A["服务实例启动"] --> B["注册中心"]
  C["调用方"] --> B
  B --> D["返回可用实例列表"]
  C --> E["调用目标服务"]
```

### API 网关

API 网关是微服务系统的统一入口。

核心功能：

- **流量路由**
  - 按路径、请求头、用户身份转发。
  - 集成负载均衡和服务发现。
- **协议适配**
  - HTTP/HTTPS 与 gRPC 等内部协议转换。
  - JSON/XML 格式转换。
  - 支持 API 版本化。
- **安全管控**
  - JWT/OAuth2 验证。
  - API Key。
  - RBAC/ABAC 权限控制。
  - HTTPS 加密。
  - 攻击过滤。
- **流量治理**
  - 限流。
  - 熔断。
  - 超时控制。
  - 灰度发布。
- **监控追踪**
  - 日志。
  - 响应时间。
  - 错误码。
  - 追踪 ID。
- **边缘处理**
  - 静态资源托管。
  - 请求签名。
  - A/B 测试。

```mermaid
flowchart LR
  A["客户端"] --> B["API 网关"]
  B --> C["用户服务"]
  B --> D["订单服务"]
  B --> E["支付服务"]
  B --> F["生产服务"]
```

### Sidecar 与服务网格

**Sidecar 模式** 是将辅助功能以独立进程与主服务部署在同一环境中。

Sidecar 负责：

- 服务发现。
- 负载均衡。
- 熔断限流。
- 监控采集。
- 日志追踪。
- 服务间通信代理。

优点：

- 剥离非业务逻辑。
- 主服务无需关心通信细节。
- 适配多语言微服务。

**服务网格（Service Mesh）** 是微服务间通信的基础设施层，由数据平面和控制平面组成。

- **数据平面**
  - 每个服务旁的 Sidecar，如 Envoy。
  - 负责流量转发、负载均衡、TLS 加密、故障处理。
- **控制平面**
  - 集中配置路由策略、安全策略、服务发现和监控。

典型工具：

- Istio。
- Linkerd。
- Consul Connect。

### Sidecar 与服务网格区别

| 维度 | Sidecar 模式 | 服务网格 |
|---|---|---|
| **定位** | 单个服务的辅助工具 | 全局通信基础设施 |
| **范围** | 一对一，服务与 Sidecar 组合 | 所有 Sidecar 构成通信网络 |
| **控制平面** | 无 | 有 |
| **功能** | 基础通信、监控等局部功能 | 流量、安全、观测性完整体系 |

```mermaid
flowchart TD
  subgraph ServiceA["服务 A Pod"]
    A["业务服务 A"] --- AS["Sidecar A"]
  end
  subgraph ServiceB["服务 B Pod"]
    B["业务服务 B"] --- BS["Sidecar B"]
  end
  AS --> BS
  C["控制平面"] -.配置策略.-> AS
  C -.配置策略.-> BS
```

## 监控、日志与故障管理

### 服务监控指标

应用层指标：

- 平均延迟。
- P95/P99 延迟。
- TPS/QPS。
- 并发量。
- 错误率。
- 服务间调用成功率。
- 熔断或限流触发次数。
- 业务指标，如订单创建量、登录频次。

基础设施指标：

- CPU 使用率。
- 内存使用率。
- 磁盘 I/O。
- 网络带宽。
- 网络延迟。
- 丢包率。
- 线程数。
- 文件句柄数。

可用性与弹性指标：

- 服务实例在线状态。
- 健康检查结果。
- 请求重试次数。

### 故障管理

主要故障类型：

- **内部逻辑错误**
- **外部依赖失效**
- **性能下降**

HTTP 状态码可用于标准化故障报告：

- `404`：资源不可访问。
- `422`：不可处理的实体，例如订单输入错误。

故障应对机制：

- **超时机制**
  - 可检测服务无响应或过慢。
  - 但超时过长会拖慢整体系统。
- **断路器模式**
  - 当服务调用多次失败后，立即阻断请求并返回故障响应。
  - 后续定期探测服务是否恢复。
  - 恢复后重置电路。

```mermaid

  [*] --> Closed: 正常调用
  Closed --> Open: 连续失败超过阈值
  Open --> HalfOpen: 定期探测
  HalfOpen --> Closed: 探测成功
  HalfOpen --> Open: 探测失败
```

## 复习要点

- 微服务应按 **业务能力** 拆分，而不是按技术层拆分。
- DDD 中，**限界上下文** 用于划定业务语义边界。
- 一个名词在不同限界上下文中可以有不同含义。
- 理想情况下，一个限界上下文对应一个微服务。
- **聚合根** 是聚合对外唯一入口，负责维护业务不变量和事务边界。
- 跨上下文协作常通过领域事件实现最终一致性。
- RESTful 适合对外接口，gRPC 适合稳定、高性能的内部通信。
- 服务注册与发现解决实例地址动态变化问题。
- API 网关是微服务统一入口，负责路由、安全、协议适配、流量治理。
- Sidecar 是单服务旁路代理，服务网格是全局通信治理体系。
- 微服务必须配套监控、日志、超时、熔断等治理机制。

## 易混点

- **限界上下文与微服务**
  - 限界上下文是业务语义边界。
  - 微服务是技术部署形态。
- **限界上下文与聚合**
  - 限界上下文解决“概念含义在哪里一致”。
  - 聚合解决“一次事务内哪些对象必须一致”。
- **高内聚不是小到不能再小**
  - 一个微服务可以包含多个相关功能，只要围绕同一业务能力。
- **RESTful 与 gRPC**
  - RESTful 灵活、易调试、适合对外。
  - gRPC 高性能、强契约、适合内部稳定服务。
- **Sidecar 与服务网格**
  - Sidecar 是局部模式。
  - 服务网格是由大量 Sidecar 加控制平面组成的系统性方案。
